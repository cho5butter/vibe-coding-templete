# プロジェクトルール — Claude Code

## 基本原則

このリポジトリは「使用駆動開発（Usage-Driven Development）」テンプレートである。
すべてのコード変更は品質ゲートを通過しなければならない。

## 厳格ルール（違反禁止）

### 1. spec管理（要件・設計・計画）

`spec/` フォルダに以下の3ドキュメントを管理する:

| ファイル | 内容 |
|----------|------|
| `spec/requirements.md` | 要件定義（ユーザーストーリー、機能要件、非機能要件） |
| `spec/design.md` | 設計（アーキテクチャ、データモデル、状態遷移、API設計） |
| `spec/plan.md` | 実装計画（セッション単位のタスク分割） |

**ルール:**
- 各ドキュメントは原則1ファイル（Markdown＋Mermaid図）で管理する
- 内容が膨大になった場合のみファイルを分割してよい（例: `spec/requirements-auth.md`）
- Mermaid図を積極的に活用し、視覚的に理解しやすくすること

### 2. 計画の粒度と実行

**1セッション＝1計画の原則:**
- `spec/plan.md` の各計画は、AIエージェントの1回のセッションで完了できるサイズに分割すること
- 計画には明確な完了条件（テスト・品質ゲートで検証可能）を含めること
- 計画が大きすぎる場合はさらに分割すること

**計画実行のワークフロー:**
1. `spec/plan.md` から次の計画を確認する
2. 計画の内容に基づいてTDDで実装する
3. 品質ゲートを実行する
4. **計画の実行後、必ずコミットを行うこと**（計画実行中に生成されたMDファイルも含める）

**spec変更時のユーザー合意:**
- 計画の実行によって `spec/requirements.md`、`spec/design.md`、`spec/plan.md` の変更が必要になった場合、**必ずユーザーに変更内容を提示し、合意を得てから変更すること**
- ユーザーの合意なくspecファイルを変更してはならない

### 3. コミット時のMDファイル包含

- コミット時、計画によって生成・変更されたMDファイル（`spec/plan.md` 含む）は**必ずコミットに含めること**
- Planファイル（計画MD）がセッション中に出力された場合、そのMDファイルもコミットに含めること
- MDファイルだけをコミットから除外してはならない

### 4. 品質ゲート（pre-commitフックで自動強制）

このリポジトリでは `hooks/pre-commit` により、コミット時に品質ゲートが**自動実行**される。
品質ゲートを通過しない限り、gitがコミットを拒否する。

**初回セットアップ（リポジトリをクローンした直後に1回実行）:**

```bash
bash scripts/setup-hooks.sh
```

**品質ゲートの内容（順に実行、1つでも失敗でコミット拒否）:**
1. `scripts/lint.sh` — リント・静的解析
2. `scripts/build.sh` — ビルド確認
3. `scripts/test.sh` — 単体テスト・結合テスト

手動で事前確認する場合:

```bash
bash scripts/quality-gate.sh
```

**`--no-verify` でのフックスキップは禁止。** 品質ゲートを回避するコミットは許されない。

### 5. TDD（テスト駆動開発）

- 新機能を実装する前に、まずテストを書くこと
- テストが失敗することを確認してから実装を開始すること
- テストが通るまでコードを修正し続けること
- テストなしのコードをコミットしてはならない

### 6. コミットメッセージ規約

コミットメッセージは**日本語**で記述すること:

```
<種別>: <変更内容の要約>

<詳細な説明（必要に応じて）>
```

種別:
- `機能`: 新機能の追加
- `修正`: バグ修正
- `改善`: 既存機能の改善
- `整理`: リファクタリング・コード整理
- `テスト`: テストの追加・修正
- `文書`: ドキュメントの変更
- `設定`: 設定ファイルの変更
- `計画`: 計画の追加・更新

例: `機能: ユーザー認証のログイン機能を追加`

### 7. PR作成ルール

PRを作成する際は以下のテンプレートに従うこと（`.github/PULL_REQUEST_TEMPLATE.md` 参照）:
- 概要セクション必須
- チェックリスト全項目を確認
- テスト計画を記載

### 8. コード品質

- 既存ファイルの編集を優先し、不要なファイル作成を避ける
- セキュリティ脆弱性（インジェクション、XSS等）を絶対に入れない
- 過剰設計をしない。現在の要件に必要な最小限の複雑さに留める
- 未使用のコード・変数・インポートを残さない

### 9. ワークフロー

1. `spec/plan.md` から次の計画を確認する
2. 要件（`spec/requirements.md`）と設計（`spec/design.md`）を参照する
3. テストを先に書く（TDD）
4. 最小限の実装を行う
5. `scripts/quality-gate.sh` を実行
6. すべてパスしたらコミット（**MDファイルを含める**）
7. 失敗したら修正して再実行
8. specの変更が必要ならユーザーに合意を得て更新

## プロジェクト固有設定

テスト・リント・ビルドのコマンドは `scripts/` ディレクトリ内のスクリプトに定義されている。
新しいプロジェクトを開始する際は、各スクリプト内の `TODO` コメントを書き換えること。
