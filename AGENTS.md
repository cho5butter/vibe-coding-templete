# プロジェクトルール — OpenAI Codex

## 基本原則

このリポジトリは「使用駆動開発（Usage-Driven Development）」テンプレートである。
すべてのコード変更は品質ゲートを通過しなければならない。

## 厳格ルール（違反禁止）

### 1. 品質ゲート

コードを変更したら、コミット前に必ず品質ゲートを実行すること:

```bash
bash scripts/quality-gate.sh
```

品質ゲートは以下を順に実行する:
1. `scripts/lint.sh` — リント・フォーマット
2. `scripts/build.sh` — ビルド確認
3. `scripts/test.sh` — テスト実行

**すべてパスしない限り、絶対にコミットしてはならない。**

### 2. TDD（テスト駆動開発）

- 新機能を実装する前に、まずテストを書くこと
- テストが失敗することを確認してから実装を開始すること
- テストが通るまでコードを修正し続けること
- テストなしのコードをコミットしてはならない

### 3. コミットメッセージ規約

コミットメッセージは**日本語**で記述すること:

```
<種別>: <変更内容の要約>

<詳細な説明（必要に応じて）>
```

種別:
- `機能`: 新機能の追加
- `修正`: バグ修正
- `改善`: 既存機能の改善
- `整理`: リファクタリング・コード整理
- `テスト`: テストの追加・修正
- `文書`: ドキュメントの変更
- `設定`: 設定ファイルの変更

例: `機能: ユーザー認証のログイン機能を追加`

### 4. PR作成ルール

PRを作成する際は `.github/PULL_REQUEST_TEMPLATE.md` のテンプレートに従うこと。

### 5. コード品質

- 既存ファイルの編集を優先し、不要なファイル作成を避ける
- セキュリティ脆弱性を絶対に入れない
- 過剰設計をしない
- 未使用のコード・変数・インポートを残さない

### 6. ワークフロー

1. 要件を理解する
2. テストを先に書く（TDD）
3. 最小限の実装を行う
4. `bash scripts/quality-gate.sh` を実行
5. すべてパスしたらコミット
6. 失敗したら修正して再実行
