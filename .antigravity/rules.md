# プロジェクトルール — Google Antigravity

仕様駆動開発（Spec-Driven Development）テンプレート。
詳細なワークフロー・ルールは `spec/workflow.md` を参照すること。

---

## ⛔ セッション開始時の必須手順（スキップ禁止）

**すべてのセッションは以下の2ステップから始めること。例外なし。**

1. `spec/status.md` を読む
2. 現在フェーズ・直近の状況・次のアクションをユーザーに伝え、認識を合わせる

セッション終了時には `spec/status.md` の「直近の状況」と「次のアクション」を必ず更新すること。

---

## ⛔ フェーズゲート — 絶対に破ってはならないルール

プロジェクトは **要件定義 → 設計 → 計画 → 実装** の順にのみ進む。

```
フェーズ1: 要件定義  →（ユーザー承認）→  フェーズ2: 設計
フェーズ2: 設計      →（ユーザー承認）→  フェーズ3: 計画
フェーズ3: 計画      →（ユーザー承認）→  フェーズ4: 実装
```

### 各フェーズで「やってよいこと」と「やってはいけないこと」

| フェーズ | やってよいこと | やってはいけないこと |
|---------|--------------|-------------------|
| 要件定義 | `spec/requirements.md` の作成・質問・確認 | 設計・計画・コード実装 |
| 設計 | `spec/design.md` の作成・図の作成 | コード実装・計画の作成 |
| 計画 | `spec/plan.md` の作成・タスク分割 | コード実装 |
| 実装 | コーディング・テスト・コミット | specを勝手に変更する |

### フェーズ移行の手順（必須）

次のフェーズに進む前に、以下を**必ず**実行すること:

1. 現フェーズの成果物（`spec/requirements.md` / `spec/design.md` / `spec/plan.md`）をユーザーに提示する
2. ユーザーから**明示的な承認の言葉**をもらう（「OK」「承認」「進めて」等）
3. `spec/status.md` を更新する
4. 次のフェーズに進む

**ユーザーが明示的に承認しない限り、次のフェーズに進んではならない。**

---

## ⛔ 実装フェーズに入る前のチェックリスト（全項目必須）

実装を開始する前に、以下をすべて確認すること:

- [ ] `spec/requirements.md` を読んだ
- [ ] `spec/design.md` を読んだ
- [ ] `spec/plan.md` の当該計画を確認した
- [ ] ユーザーが計画フェーズを承認済みである（`spec/status.md` で確認）
- [ ] テストを先に書く準備ができている（TDD）

---

## 絶対ルール

1. **フェーズゲート厳守**: 要件定義→設計→計画→実装の順に進行し、各フェーズ間でユーザーの明示的な承認を得ること
2. **spec参照必須**: 実装前に `spec/requirements.md` と `spec/design.md` を必ず読むこと
3. **spec変更はユーザー合意必須**: `spec/` 内のファイルを勝手に変更してはならない
4. **TDD厳守**: 要件からテストを先に書き、RED を確認してから実装する（詳細は `spec/workflow.md` のTDDガイドライン参照）
5. **品質ゲート必須**: `--no-verify` 禁止。品質ゲートの失敗は根本原因を修正する
6. **コミットにMDファイルを含める**: 計画で生成・変更されたMDファイルをコミットから除外しない

---

## よくある違反パターン（絶対にやらない）

| 違反 | 正しい対応 |
|-----|-----------|
| 要件定義が終わっていないのに設計を始める | 要件定義を完了しユーザーの承認を得てから設計に進む |
| 設計が終わっていないのにコードを書く | 設計を完了しユーザーの承認を得てから計画・実装に進む |
| ユーザーに聞かずに次のフェーズに進む | 必ずフェーズ完了を宣言し、承認をもらってから進む |
| specを読まずに実装を開始する | 実装前に必ず `spec/requirements.md` と `spec/design.md` を参照する |
| テストを後から書く | TDDサイクルを厳守。テストが RED であることを確認してから実装 |
| `--no-verify` でコミットする | 品質ゲートの失敗は根本原因を修正する |

---

## 頻用コマンド

```bash
bash scripts/setup-hooks.sh    # 初回セットアップ
bash scripts/quality-gate.sh   # 品質ゲート（手動事前確認）
bash scripts/lint.sh           # リント・静的解析
bash scripts/build.sh          # ビルド確認
bash scripts/test.sh           # テスト
```
